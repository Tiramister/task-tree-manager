## Context

現在のアプリケーションは `max-w-3xl` の固定幅コンテナで構成されており、デスクトップでの利用を前提としている。future-spec.md ではレスポンシブデザイン（スマホブラウザでも快適に動作）が技術要件として定められている。Tailwind CSS が既に導入済みで、レスポンシブユーティリティクラスが利用可能。

## Goals / Non-Goals

**Goals:**
- モバイル（~640px）・タブレット（~1024px）・デスクトップで適切なレイアウトを提供する
- タッチ操作に最適化されたUI（十分なタップターゲットサイズ、スワイプ操作）
- 詳細ドロワーのモバイル対応（モバイルでは全画面 Sheet）

**Non-Goals:**
- ネイティブアプリ対応（PWA化は将来検討）
- ドラッグ&ドロップによるタスク並べ替え（Phase 6 のスコープ外）
- オフライン対応

## Decisions

### 1. ブレークポイント戦略：Tailwind CSS のデフォルトブレークポイントを使用
- **選定**: `sm`(640px), `md`(768px), `lg`(1024px) を活用
- **理由**: Tailwind の標準ブレークポイントで十分。カスタム定義は不要
- **代替案**: カスタムブレークポイント → 標準で問題ないため不採用

### 2. ドロワーのモバイル対応：Shadcn/ui Sheet のレスポンシブ制御
- **選定**: CSS メディアクエリでドロワー幅を制御。モバイルでは全幅表示
- **理由**: Shadcn/ui Sheet（Radix UI ベース）はそのまま利用でき、幅の調整のみで対応可能
- **代替案**: モバイル用の別ページ遷移 → SPA の利点を活かすためドロワー方式を維持

### 3. タッチ最適化：CSS ベースのアプローチ
- **選定**: タップターゲット最小44px、`touch-action` の適切な設定、hover 状態の代替
- **理由**: 追加ライブラリ不要で Tailwind のユーティリティクラスで実現可能
- **代替案**: タッチ専用ライブラリ → 現時点では過剰

### 4. モバイルレイアウト：コンテナ幅の動的調整
- **選定**: モバイルではパディングを縮小し画面幅を最大限活用。ツリーのインデントも縮小
- **理由**: 狭い画面でコンテンツ領域を確保するため

## Risks / Trade-offs

- **ツリー構造の深いネストがモバイルで見づらくなる** → インデント幅を縮小（デスクトップ: 24px → モバイル: 16px）で対応
- **タッチとマウス操作の両立** → `@media (hover: hover)` でホバー状態を制御し、タッチデバイスでは不要なホバー効果を抑制
