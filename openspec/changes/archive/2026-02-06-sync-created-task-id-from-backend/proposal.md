## Why

ログイン済み状態で新規タスクを作成すると、バックエンドでは新しいタスク ID が採番される一方、フロントエンド側のタスク ID がローカル生成値のまま残るケースがある。ID の不一致は後続の更新・削除・親子参照の整合性を崩すため、作成直後にフロントエンド状態をバックエンド採番 ID に一致させる必要がある。

## What Changes

- 新規タスク作成時、`POST /tasks` の成功レスポンスに含まれるタスク ID をフロントエンドの該当タスクへ反映する要件を追加する。
- ID 反映時に、作成直後タスクを参照するフロントエンド状態（親子関係や後続操作対象）が一貫して新 ID を使うことを要件化する。
- バックエンド同期時の「レスポンスで全体上書きしない」という方針を維持しつつ、作成結果に限った最小更新を許容するよう要件を明確化する。

## Capabilities

### New Capabilities
- なし

### Modified Capabilities
- `task-sync`: タスク作成 API 成功時に、バックエンド採番 ID をフロントエンド側タスクへ確定反映する要件を追加する。

## Impact

- フロントエンドの同期処理（task sync service）
- フロントエンドのタスク状態管理（task store における作成直後タスクの ID 更新）
- タスク作成後に続く更新・削除・並べ替えなど ID 参照を伴う処理の整合性
